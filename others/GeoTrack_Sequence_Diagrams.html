GeoTrack — Diagrammes de Séquence UML

Modélisation comportementale des 5 processus métier critiques de la plateforme par appels asynchrones (WebSocket) et synchrones (REST API).

1

Suivi GPS en Temps Réel (Core)

Processus continu d'envoi des coordonnées, d'enregistrement en base et de diffusion WebSocket.

sequenceDiagram autonumber actor D as Traceur GPS (Simulateur) participant API as API REST (/api/track) participant DB as PostgreSQL (Knex) participant WS as Serveur WebSocket actor C as Client React (Opérateur) C->>WS: Connexion WebSocket (auth JWT) WS-->>C: Connecté (Room entreprise) rect rgb(248, 250, 252) note right of D: Boucle asynchrone (toutes les 5s) D->>+API: POST /api/track {lat, lng, speed} API->>+DB: UPDATE devices SET position DB-->>-API: Succès (1 row) API->>+DB: INSERT INTO device_history DB-->>-API: Succès (1 row) API-->>WS: emit('device-moved', payload) deactivate API WS-->>C: Broadcast: event 'device-moved' C->>C: Zustand Store: updateDevice() C->>C: Leaflet Map: Animer marqueur end
2

Détection et Notification d'une Alerte (Geofencing)

Calcul géospatial, déclenchement d'alerte métier et notification multi-canal (WS + Email).

sequenceDiagram autonumber participant Engine as Moteur Simulation participant GeoLogic as Logique Geofence participant DB as PostgreSQL participant WS as Serveur WebSocket participant Mail as API Mailjet (SMTP) Engine->>+DB: SELECT zones FROM geofences WHERE enterprise_id DB-->>-Engine: Polygons & Cercles Engine->>+GeoLogic: checkIntersection(position_gps, zones) GeoLogic-->>-Engine: RESULTAT: Sortie de zone (EXIT) Engine->>+DB: INSERT INTO alerts (type: 'geofence', severity: 'high') DB-->>-Engine: Alerte créée {id} par Push Temps Réel Temps Engine-->>WS: emit('new-alert', alert_data) WS-->>Client React: Affichage d'un toast notification (Popup) and Push Email Asynchrone Engine->>+DB: SELECT emails FROM notification_rules WHERE type='geofence' DB-->>-Engine: Liste de diffusion [array] Engine->>+Mail: sendEmail(template_alerte, destinataires) Mail-->>-Engine: HTTP 200 OK (Email expédié) end
3

Interaction Chatbot IA Contextuel

Passage d'une question utilisateur via l'API, concaténation du contexte base de données et orchestration n8n vers Ollama.

sequenceDiagram autonumber actor Admin participant React as React (Dashboard) participant API as API (/api/dashboard-chat) participant DB as PostgreSQL participant n8n as Serveur n8n (Webhook) participant Ollama as Modèle Ollama local Admin->>React: Pose question "Appareils hors ligne ?" React->>+API: POST {message: "..."} API->>+DB: SELECT COUNT(*) FROM devices WHERE status='offline' DB-->>-API: Résultat {count: 12} API->>+DB: SELECT données globales (Alerte, Commandes) DB-->>-API: Contexte riche jsonb API->>API: Préparation System Prompt injecté avec contexte API->>+n8n: POST vers Webhook {system_prompt, user_msg} n8n->>+Ollama: Requête LLM Inférence Ollama-->>-n8n: "Il y a actuellement 12 traceurs hors ligne." n8n-->>-API: Réponse formatée API-->>-React: 200 OK {response: "..."} React-->>Admin: Affichage bulle chat IA
4

Workflow de Traitement de Commande B2B

Processus administratif déclenchant la génération massive de ressources en cascade (compte, entreprise, appareils).

sequenceDiagram autonumber actor Admin participant React as Panel Admin participant API as API /api/orders participant DB as PostgreSQL participant Mail as API Mailjet Admin->>React: Clique "Activer commande N°123" React->>+API: PATCH /api/orders/123/status {status: 'active'} API->>+DB: UPDATE orders SET status='active' WHERE id=123 DB-->>-API: OK API->>+DB: INSERT INTO enterprises (name, plan) DB-->>-API: Retourne enterprise_id API->>+DB: INSERT INTO users (role='operator', hash) DB-->>-API: Retourne user_id loop Pour N appareils commandés API->>API: Génération auto IMEI & Numéro de série (ex: GT-0001) API->>+DB: INSERT INTO devices (enterprise_id, imei) DB-->>-API: OK end API->>+Mail: sendWelcomeEmail(credentials, login_link) Mail-->>-API: 200 OK API-->>-React: 200 OK (Activation réussie) React-->>Admin: Rafraichissement UI, statut "Actif"
5

Authentification JWT & Contrôle d'Accès (RBAC)

Délivrance de Token crypté, vérification sur Middleware et Scope au niveau BDD.

sequenceDiagram autonumber actor User as Opérateur participant UI as React Frontend participant Auth as Auth Controller participant Mid as Middleware JWT (requireRole) participant Res as Ressource (Geofences) participant DB as PostgreSQL rect rgb(250, 250, 250) note right of User: Phase 1 : Login User->>UI: Saisie email + mot de passe UI->>+Auth: POST /api/auth/login Auth->>+DB: SELECT * FROM users WHERE email DB-->>-Auth: row.hash (bcrypt) Auth->>Auth: valid = bcrypt.compare(pass, hash) Auth->>Auth: token = jwt.sign({id, role, enterpriseId}) Auth-->>-UI: 200 OK {token} UI->>UI: Zustand: setToken(token) en mémoire end rect rgb(240, 253, 244) note right of User: Phase 2 : Accès Restreint User->>UI: Clique "Geofences (Plan Pro)" UI->>UI: PlanRoute Check: user.plan == "pro" ? (OUI) UI->>+Mid: GET /api/geofences (Header: Bearer token) Mid->>Mid: jwt.verify() -> Décodage valide Mid->>Mid: checkRole(['admin', 'operator']) -> Opérateur OK Mid->>+Res: next() -> Lance le Contrôleur Res->>+DB: SELECT * FROM geofences WHERE enterprise_id DB-->>-Res: Zones appartenant à l'entreprise Res-->>-Mid: JSON Mid-->>-UI: 200 OK UI-->>User: Affiche l'interface Geofences end
Information Technique : Ces diagrammes UML de séquence sont générés dynamiquement par Mermaid.js au chargement de la page. Les lignes continues (->>) représentent les appels synchrones (HTTP/Calculs) et les lignes pointillées (-->>) représentent les réponses ou envois asynchrones (WebSocket, Promises).
